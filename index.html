Splunk SIEM — MFA & Access Control Compliance Dashboard

Using Splunk and the BOTSv3 dataset to detect multi-factor authentication policy violations inside a simulated financial services company.

Splunk SIEM BOTSv3 Dataset MFA Compliance Access Control SPL Queries Security Dashboards

Dulneth Kurunduwatte Gedara  ·  2024

01 — Overview

The scenario

Sutrefia (Sunken Treasure Financials Australia) is a mid-to-large financial services company offering banking, investments, and insurance. As a prime target for cyber threats, the organisation has developed a suite of security policies aligned with the Australian Government Information Security Manual (ISM).

Following an internal security incident, management evaluated Splunk as their SIEM (Security Information and Event Management) tool. My task was to build a dashboard using the BOTSv3 sample dataset that gives non-technical managers a clear, at-a-glance view of whether security policies are being followed.

Policy under review: Access Control & MFA — All systems should require multi-factor authentication (MFA) to access them, including internal systems.

I focused on this policy because access control is the first line of defence in any financial organisation. Even the most sophisticated security infrastructure can be undermined if users can bypass MFA. The BOTSv3 dataset provided rich authentication log data across AWS and Azure AD environments to make this analysis concrete.

02 — Dashboard Design

Three key indicators

Rather than building a generic log viewer, I designed the dashboard around three specific indicators — each one tells a different part of the MFA compliance story.

01
Login access incidents where MFA was not followed

This panel identifies users who gained system access without completing MFA verification. It pulls from the BOTSv3 AWS source and flags sessions where mfaAuthenticated = false, showing the user, the application accessed, and both the first and last failed attempt timestamps.

637 web_admin bypasses
611 bstoll bypasses
73 btun bypasses
45 splunk_access bypasses
⚠

web_admin bypassed MFA 637 times — the highest of any account. Given this is an admin-level account with elevated privileges, this represents a critical risk. A compromised admin account without MFA is a complete system-level threat.

// SPL query

SPL — Indicator 1 
index=botsv3 sourcetype=aws* earliest=0
| search "userIdentity.sessionContext.attributes.mfaAuthenticated"=false
| stats count as "Failed MFA Logins",
        values(userAgent) as "Applications",
        earliest(_time) as "First Attempt",
        latest(_time) as "Last Attempt"
        by userIdentity.userName
| sort - "Failed MFA Logins"
| head 20
| rename userIdentity.userName as "User",
        "Failed MFA Logins" as "MFA Failed Count",
        "First Attempt" as "First Failed Attempt",
        "Last Attempt" as "Last Failed Attempt"
| eval "First Failed Attempt"=strftime('First Failed Attempt', "%Y-%m-%d %H:%M:%S"),
      "Last Failed Attempt"=strftime('Last Failed Attempt', "%Y-%m-%d %H:%M:%S")
| table User, "MFA Failed Count", "Applications", "First Failed Attempt", "Last Failed Attempt"
02
Patterns and frequencies of authentication failures

This panel focuses on failed login patterns across multiple application sources — AWS, Azure AD (ms:aad), Windows Security Event logs, and WinEventLog. By aggregating these into a single view with failure reasons and timestamps, management can spot brute-force attempts, account lockouts, and insider threats at a glance.

A high concentration of failures from a single user or at unusual hours is a red flag — this panel makes that visible without requiring technical expertise.

// SPL query

SPL — Indicator 2 
index=botsv3 earliest=0
(sourcetype="aws*" AND eventName="SigninFailure")
OR (sourcetype="XmlWinEventLog:Microsoft-Windows-Security-Auditing"
    AND EventCode=4625)
OR (sourcetype="ms:aad:signin" AND loginStatus="Failure")
OR (sourcetype="wineventlog:security" AND EventCode=4625)

| eval Time=strftime(_time, "%Y-%m-%d %H:%M:%S")
| eval Application=coalesce(appDisplayName, source, sourcetype)
| eval User=coalesce(userIdentity.userName, userPrincipalName)
| eval Reason=coalesce(errorMessage, failureReason, status)
| rename Application as Application_Name, User as User,
         Reason as Reason_of_Failure, Time as Occur_Time
| table Application_Name, User, Reason_of_Failure, Occur_Time
| sort - Occur_Time
| head 10
03
MFA enabled vs disabled users & session logout monitoring

This dual-purpose panel answers two questions management really care about: how many users have MFA turned on? and are users logging out properly?

The comparison table and pie chart pull from Azure AD sign-in data to show MFA adoption rates across the organisation. The logout events panel identifies sessions where a user logged in successfully via MFA but never formally logged out — leaving an open session that anyone with physical or remote access could hijack.

// SPL query — MFA enabled vs disabled

SPL — Indicator 3a 
index=botsv3 sourcetype=ms:aad:signin
| eval MFA_Status = if(isnull(mfaRequired)
    OR mfaRequired="false", "MFA Not Enabled", "MFA Enabled")
| stats count as signInCount
    by userDisplayName, userPrincipalName, MFA_Status
| sort - signInCount
| head 10
| table userDisplayName, userPrincipalName, MFA_Status, signInCount

// SPL query — MFA success & logout events

SPL — Indicator 3b 
index=botsv3 sourcetype="wineventlog:security"
| eval event_type=case(
    EventCode=4624, "MFA Success",
    EventCode=4634, "Logout")
| stats count by Account_Name, event_type, EventCode, _time
| eventstats values(_time) as login_time by Account_Name
| eval logged_out=if(
    event_type="Logout" AND _time > login_time, "Yes", "No")
| where event_type="MFA Success" OR event_type="Logout"
| sort Account_Name, _time
| dedup Account_Name
| head 10
| table _time, Account_Name, event_type, logged_out
03 — Key Findings

What the data revealed

Together, the three indicators paint a clear picture of widespread MFA non-compliance within the simulated Sutrefia environment.

1,366+ Total MFA bypasses
4 Admin accounts bypassing MFA
~70% Users with MFA disabled
0 Users logging out properly

// Critical: admin-level bypass concentration

The web_admin account had 637 MFA bypasses — more than all other accounts combined. Admin accounts have the highest privilege level and represent the greatest risk if compromised. This single finding alone justifies immediate MFA enforcement on all admin accounts.

// Session abandonment risk

Across all monitored accounts, no users logged out properly after their MFA-authenticated sessions. Open sessions on unattended workstations are a common attack vector — particularly concerning in a financial services environment with access to sensitive transaction data.

04 — Conclusion

Why this dashboard matters

This Splunk dashboard gives management a real-time, evidence-based view of MFA policy compliance — without needing to understand the technical details of log analysis. Each panel is designed to answer a specific management question:

Panel 1 answers: "Who is getting in without MFA, and how often?"
Panel 2 answers: "Are there patterns in login failures that suggest attack attempts?"
Panel 3 answers: "How many of our users actually have MFA set up, and are they using it properly?"

The BOTSv3 data shows that MFA policy non-compliance is not an edge case — it is the norm within this environment. The highest-risk accounts (admin-level) have the worst compliance. Without a monitoring tool like this dashboard, these violations would go undetected until an incident occurs.

Implementing this dashboard as a live monitoring tool, with alerts triggered on thresholds (e.g., more than 10 MFA bypasses from a single account in one hour), would significantly reduce response time and improve Sutrefia's overall security posture.
